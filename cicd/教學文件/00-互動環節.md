## 互動 1：快問快答（5 min）

> **時機**：在 Demo 1（第一個 Workflow）之後

### 題目 1（暖身題）
**Q：CI 和 CD 最大的差別是什麼？**

<details>
<summary>答案</summary>

- **CI（持續整合）**：專注在「程式碼合併後自動測試」，確保程式碼品質
- **CD（持續交付/部署）**：專注在「測試通過後自動打包和部署」，加速上線

簡單記法：CI 管「品質」，CD 管「速度」。

</details>

### 題目 2
**Q：GitHub Actions 的 Workflow 檔案要放在哪裡？**

選項：
- A. 專案根目錄
- B. `.github/actions/`
- C. `.github/workflows/`
- D. `workflows/`

<details>
<summary>答案</summary>

**C**。Workflow YAML 檔案必須放在 `.github/workflows/` 目錄下，GitHub 才會自動偵測並執行。

</details>

---

### 題目 3
**Q：在 GitHub Actions 的 Workflow 中，`runs-on: ubuntu-latest` 是什麼意思？**

<details>
<summary>答案</summary>

指定這個 Job 要在 **GitHub 提供的最新版 Ubuntu Linux 機器**上執行。這台機器叫做 **Runner**，每次執行都是全新的環境（用完即丟）。

也可以選 `windows-latest` 或 `macos-latest`。

</details>

---

### 題目 4（加分題）
**Q：`uses: actions/checkout@v4` 在做什麼？如果不加這一步會怎樣？**

<details>
<summary>答案</summary>

`actions/checkout@v4` 會把你的 repo 的程式碼 clone 到 Runner 上。

如果不加這一步，Runner 上是**完全空白**的——沒有你的程式碼、沒有 Dockerfile，什麼都沒有。後面的 `npm install`、`docker build` 都會失敗。

</details>

## 互動 2：情境題挑戰（5 min）

> **時機**：在 Demo 2（自動測試 + Build）之後

### 題目 1
**情境：你的同事在 PR 裡改了一行 code，但 CI 測試失敗了。他說：「這只是小改動，不用管測試，直接 merge 就好。」你覺得應該怎麼回應？**

<details>
<summary>答案</summary>

**不應該直接 merge！**

理由：
1. CI 存在的意義就是在合併前抓出問題
2. 「小改動」也可能造成大問題——這正是自動化測試要幫你發現的
3. 正確做法：先修好測試失敗的問題，再 merge

最佳實踐：在 GitHub 設定 **Branch Protection Rule**，要求 CI 通過才能 merge。

</details>

---

### 題目 2
**情境：你的 CI Pipeline 需要登入 Docker Hub，以下哪種做法最安全？**

- A. 把帳號密碼直接寫在 workflow YAML 檔裡
- B. 把帳號密碼寫在 `.env` 檔裡，然後 commit 到 repo
- C. 用 GitHub Secrets 儲存
- D. 寫在 README 裡面讓大家都能用

<details>
<summary>答案</summary>

**C。** 用 GitHub Secrets 儲存！

- A 和 B 都會把密碼暴露在 repo 裡（git 歷史記錄永遠都在）
- D 不用說了... 😂
- Secrets 的值不會出現在 log 中，也不會被 fork 的 repo 看到

</details>

### 題目 3
**Q：在以下的 workflow 中，如果 `test` Job 失敗了，`build` Job 會執行嗎？**

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: exit 1    # 模擬測試失敗

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."
```

<details>
<summary>答案</summary>

**不會！** 因為 `build` 有 `needs: test`，表示 `build` 依賴 `test`。

`test` 失敗後，`build` 會被自動跳過（顯示灰色，狀態為 "Skipped"）。

這就是 Pipeline 的安全機制——前面失敗了，後面就不用跑了。

</details>

## 互動 3：實戰判斷題（5 min）

> **時機**：在 Demo 3（完整 Pipeline）之後

### 題目 1
**Q：你的 CI/CD Pipeline 每次 push 都會 build 並 push Docker image，tag 都是 `latest`。這樣有什麼問題？**

<details>
<summary>答案</summary>

問題：
1. **無法追蹤版本**：不知道 `latest` 對應的是哪次 commit
2. **無法回滾**：想回到上一個版本，但 `latest` 已經被覆蓋了
3. **快取問題**：有些環境會快取 `latest`，不會自動拉新版

解決方案：
- 除了 `latest`，同時加上 **git SHA** 或 **語意版本** 的 tag
- 例如：`my-app:latest` + `my-app:abc123f` + `my-app:v1.2.3`

</details>

---

### 題目 2
**Q：以下兩種寫法有什麼差別？什麼時候用哪種？**

```yaml
# 寫法 A
on:
  push:
    branches: [main]

# 寫法 B
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
```

<details>
<summary>答案</summary>

- **寫法 A**：只在 push 到 main 時觸發
- **寫法 B**：push 到 main **或** 建立/更新 PR 到 main 時都會觸發

常見做法：
- **跑測試**用寫法 B → PR 階段就能發現問題
- **Build + Push image**加 `if` 條件 → 只在 push（merge 後）才 build，PR 階段只跑測試

</details>
