## 各段落詳細教學指引

---

### 1. 開場（10 min）

**目標**：承接 Docker 課程，說明「有了 Docker 之後的下一步」。

**過場引言**：
> 「上一堂課我們學了 Docker，可以把應用打包成 Image，用 Compose 管理多容器。但問題來了——每次改 code 都要手動 build、手動 push、手動部署... 能不能讓這些事情全自動？這就是今天要學的 CI/CD。」

**講解重點**：
1. 回顧 Docker 的價值：打包一次、到處執行
2. 痛點：手動 build + push + deploy 太慢太容易出錯
3. CI/CD 的核心價值：**自動化整條交付流程**
4. 今天的目標：push code → 自動測試 → 自動 build image → 自動 push 到 Docker Hub

**參考教材**：01-CI-CD基礎概念.md — 「什麼是 CI/CD？」、「為什麼需要 CI/CD？」

---

### 2. 第一段：CI/CD 核心概念（20 min）

**目標**：建立 CI/CD Pipeline 的心智模型。

**過場引言**：
> 「在動手之前，我們先把核心觀念講清楚。」

**講解重點**：

**(a) CI vs CD（8 min）**
- CI = 持續整合：每次 push code → 自動跑測試
- CD = 持續交付 / 部署：測試通過 → 自動準備 / 部署
- 畫出 Pipeline 流水線圖

**(b) Pipeline 各階段（7 min）**
- Source → Build → Test → Package → Deploy
- 每個階段的作用和範例

**(c) GitHub Actions 基本概念（5 min）**
- Workflow、Job、Step、Action 的關係
- 用食譜類比（Workflow = 食譜，Job = 一道菜，Step = 步驟）

**參考教材**：01-CI-CD基礎概念.md、02-GitHub-Actions基礎.md

---

### 3. Demo 1：第一個 Workflow（15 min）

**目標**：讓學員看到「push code 後自動跑東西」的效果。

**步驟**：

```bash
# 1. 建立 workflow 檔案
mkdir -p .github/workflows

# 2. 寫一個最簡單的 workflow
cat > .github/workflows/hello.yml << 'EOF'
name: Hello CI

on: [push]

jobs:
  say-hello:
    runs-on: ubuntu-latest
    steps:
      - name: Say Hello
        run: echo "Hello, CI/CD!"

      - name: Show date
        run: date

      - name: Show system info
        run: uname -a
EOF

# 3. Push 到 GitHub
git add .
git commit -m "Add first workflow"
git push
```

**講解要點**：
- 「這個 YAML 檔就是在告訴 GitHub：每次 push 的時候，幫我在一台乾淨的 Ubuntu 機器上跑這三個指令」
- 切到 GitHub Actions 頁面，即時看執行結果
- 點進去看每個 Step 的 log 輸出
- 強調 Runner 每次都是全新的環境

### 5. 第二段：GitHub Actions 進階（20 min）

**過場引言**：
> 「剛剛的 workflow 只是 echo 幾行字。接下來我們要做真正有用的事——跑測試和 build Docker image。」

**講解重點**：

**(a) 多 Job 與 needs（7 min）**
- 平行 vs 依序執行
- `needs` 關鍵字
- 「先跑測試，測試通過才 build image」

**(b) Secrets 管理（7 min）**
- 為什麼不能把密碼寫在 YAML 裡
- 如何設定 GitHub Secrets
- `${{ secrets.XXX }}` 語法

**(c) 條件執行（6 min）**
- `if` 語法
- PR 只跑測試，push 到 main 才 build image
- 這樣可以省 CI 時間和資源

**參考教材**：02-GitHub-Actions基礎.md

### 6. Demo 2：自動測試 + Build Image（20 min）

**目標**：展示一個包含測試和 Docker build 的 Workflow。

**步驟**：

1. 先展示 Demo 用的簡單 Node.js app 和測試
2. 加入 CI workflow，包含 `npm test` 和 `docker build`
3. Push 後看 GitHub Actions 跑兩個 Job

**講解要點**：
- 強調 `needs: test` 的效果——test 失敗就不會 build
- 故意讓一個測試失敗，展示失敗時的 UI 和通知
- 修好後再 push，展示成功的流程

### 8. 第三段：Docker Hub 整合（15 min）

**過場引言**：
> 「現在我們的 CI 會自動跑測試、自動 build image。最後一步——把 image 自動 push 到 Docker Hub，讓任何人都可以直接 pull 來用。」

**講解重點**：

**(a) Docker Hub Access Token（5 min）**
- 為什麼用 Token 不用密碼
- 現場展示如何在 Docker Hub 產生 Token

**(b) 官方 Docker Actions（5 min）**
- `docker/login-action`
- `docker/build-push-action`
- 比自己寫 `docker build` + `docker push` 指令更好

**(c) Image Tag 策略（5 min）**
- `latest` vs git SHA vs 語意版本
- 為什麼正式環境不該用 `latest`

**參考教材**：03-Docker整合GitHub-Actions.md

### 9. Demo 3：完整 CI/CD Pipeline（25 min）

**目標**：串起完整流程：push → test → build → push to Docker Hub。

**這是整堂課的高潮！**

**步驟**：

1. 展示完整的 workflow YAML（含 test + build-and-push 兩個 Job）
2. 設定 Secrets（如果還沒設定，現場展示）
3. Push code，即時觀看 Pipeline 執行
4. 最後打開 Docker Hub，展示新的 image 已經出現

**講解要點**：
- 從頭走一遍：「我只是 push code，後面的事情全部自動完成」
- 在 Docker Hub 上展示 image 的 tag
- 示範 `docker pull` 拉下剛剛自動 build 的 image
- 「這就是 CI/CD 的威力——從 push 到上架，全程不需要人工介入」

## 講師備忘

### 節奏控制
- 每 20 分鐘穿插一次互動或 Demo，避免純講太久
- 如果 Demo 時 GitHub Actions 跑太慢，可以先講後面的內容，等跑完再回來看結果
- 準備一個「已經跑完」的備用結果截圖，以防網路問題

### 常見學員問題
- 「GitHub Actions 要錢嗎？」→ 公開 repo 免費，私有 repo 每月 2000 分鐘免費
- 「公司用 GitLab 怎麼辦？」→ 概念相通，GitLab CI 語法不同但邏輯一樣
- 「可以部署到 AWS 嗎？」→ 可以，但超出今天範圍，可以自行延伸學習
